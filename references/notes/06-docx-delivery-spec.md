# DOCX 交付规范

## 目标

当用户明确要求文件交付时，skill 应直接生成一份可投递的 `.docx` 简历，而不是只返回文本内容。

## 标准流程

1. 完成岗位判断、内容诊断和经历改写；
2. 按本文件与 `references/notes/07-docx-layout-spec.md` 的规则组织内容；
3. 将最终正文映射为固定 JSON 结构；
4. 使用 `python3 scripts/generate_resume_docx.py --input-json <数据文件> --output <输出路径>` 生成 `.docx`；
5. 使用 `python3 scripts/verify_resume_docx.py <输出路径>` 做结构检查；如环境支持，再补充渲染检查；
6. 交付 `.docx` 文件。

正文到 JSON 的映射规则优先参考：

- `references/notes/12-structured-json-spec.md`
- `assets/resume-template.json`

## 默认版式原则

- 页面为 A4 纵向；
- 页边距约 1 cm，可小幅调整；
- 中文默认使用楷体风格字体；macOS 默认 `STKaiti`，Windows 可用“楷体”；
- 英文默认使用 `Calibri`；
- 姓名约 16 pt；
- 联系方式约 12 pt；
- 模块标题约 12 pt；
- 正文约 11 pt，可在 10-12 pt 范围内微调；
- 标题下方可配细横线；
- 日期右对齐；
- 以单页交付为优先目标。

## 视觉控制要求

- 版式应保持黑白、正式、紧凑；
- 视觉层级主要依赖字号、粗细、横线和缩进；
- 不要使用宣传页式装饰；
- 有效信息密度应高于装饰性效果。

## 页面适配规则

- 行间距和段间距属于可调参数；
- 内容不足一页时，可适度放松；
- 内容略超一页时，应优先压缩段前段后距、模块间距、行距或 bullet 间距；
- 但不要为了压缩而牺牲可读性。

## 文件交付时的优先级

1. 内容真实、定向、有效
2. 单页完整
3. 版式统一、专业、易读
4. 局部间距微调

## 当用户明确要求 Word 文件

应理解为：

- 生成一份本地 `.docx`；
- 内容已经按目标岗位改写；
- 版式已经按内置规范整理完成。

不要只给纯文本，不要只给 Markdown，也不要把排版责任留给用户。

## 环境受限时的处理

如果当前环境缺少完整渲染链路：

- 仍优先使用 skill 自带脚本生成 `.docx`；
- 明确说明哪些校验已执行、哪些未执行；
- 不把文件交付建立在额外第三方 Python 依赖之上。

## 对 skill 的直接要求

当用户提出以下任一需求时，应进入 Word 输出模式：

- “直接给我一份 Word”
- “帮我生成 `.docx`”
- “按这个版式排成简历文件”
- “给我一份可投递的简历文件”

此时不仅要改内容，还要完成文件级交付。
