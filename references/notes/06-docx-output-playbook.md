# 法学生简历 Word 输出打法笔记

## 目标

这个 skill 不应只输出“文字内容建议”，而应在用户明确要求时，直接产出一份可交付的 `.docx` 简历文件。

## 依赖路径

生成 `.docx` 时，优先调用 skill 自带的本地脚本 `scripts/generate_resume_docx.py`，而不是只返回 Markdown。

默认流程：

1. 先完成岗位判断、内容诊断、经历改写；
2. 再按本文件与 `references/notes/07-docx-template-structure.md` 的版式规则组织内容；
3. 先把最终正文映射成固定 JSON 结构；
4. 使用 `python3 scripts/generate_resume_docx.py --input-json <数据文件> --output <输出路径>` 创建本地 `.docx`；
5. 使用 `python3 scripts/verify_resume_docx.py <输出路径>` 做结构检查；如环境支持，再做 PDF/图片级视觉检查；
6. 最终交付 `.docx` 文件。

正文到 JSON 的规则，优先参考：

- `references/notes/12-resume-json-handoff.md`
- `assets/resume-template.json`

## 默认版式规则

输出 Word 简历时，默认遵循以下规则：

- 页面：A4 纵向
- 页边距：约 1 cm，可小幅调整
- 字体：
  - 中文默认优先楷体风格字体；macOS 默认 `STKaiti`，Windows 可用“楷体”
  - 英文优先 Calibri
- 姓名：约 16 pt
- 联系方式：约 12 pt
- 模块标题：约 12 pt，略大于正文
- 正文：约 11 pt，允许在 10-12 pt 范围内调整
- 标题下方配细横线
- 日期右对齐
- 内容整体保持单页优先

补充理解：

- 这类法学生简历默认是黑白、正式、紧凑的投递型版式；
- 视觉重点主要来自字号层级、粗细、横线和缩进，而不是颜色和装饰；
- 模块标题和正文的字号差距不应过大，更接近“正文 11 pt / 标题 12 pt / 姓名 16 pt”的克制梯度；
- 不要做成宣传页或花哨模板，重点是专业、易扫读、单页有效信息密度高。

## 弹性规则

行间距和段间距不是固定常数，应为页面适配服务：

- 只要整体仍然专业、易读、整洁，行间距和段间距都可以自由调整；
- 内容不满一页时，可以略微增大段前段后距或行距；
- 内容刚好超出一页时，应优先压缩段前段后距、模块间距、行距或 bullet 间距，直到回到一页内；
- 但不要为了塞进一页而把正文压到不可读。

换句话说：

- 单页是第一优先级之一；
- 行距不是必须固定的版式规范；
- 段距也不是必须固定的版式规范；
- 只要阅读性还在，就允许继续压缩。

## 输出文件时的优先级

从高到低：

1. 内容真实、定向、有效
2. 单页完整
3. 版式接近参考 PDF
4. 局部间距微调

不要为了复制 PDF 形式而牺牲内容准确性。

## 当用户要求“直接给我 Word”

应理解为：

- 输出一份本地 `.docx`
- 内容已经按目标岗位改写完
- 版式按参考 PDF 的风格排好

不要只给纯文本，不要只给 Markdown，不要只说“你可以复制到 Word”。

## 当环境限制导致无法完整渲染验证

如果当前环境缺少生成或渲染依赖：

- 生成路径优先依赖 skill 自带脚本，而不是额外 pip 安装第三方库；
- 仍应优先构造 `.docx` 产出路径；
- 明确说明哪个环节缺依赖；
- 但 skill 本身仍要坚持“目标产物是 `.docx`”这一要求。

## 对 skill 的直接要求

当用户说以下任一类话时，应主动进入 Word 输出模式：

- “直接给我一份 Word”
- “帮我生成 `.docx`”
- “按这个模板排成简历文件”
- “参考这份 PDF 给我做一份可投递简历”

此时不仅要改内容，还要完成文件级产出。
