# 输出行为规范

## 目标

本文件用于固定 skill 的输出顺序和信息密度，明确什么时候先分析，什么时候再出简历，什么时候进入 `.docx` 文件交付。

## 默认输出节奏

### 第一轮

在以下情况，默认先输出修改意见而不是终稿：

- 用户提供岗位要求和现有简历；
- 用户只提供个人经历、素材或零散信息；
- 用户要求分析简历问题；
- 用户要求看看简历怎么改。

第一轮必须优先完成：

1. 使用场景判断
2. 当前问题
3. 调整思路
4. 进一步内容挖掘问题
5. 下一步说明

### 第二轮

在以下情况，进入定向简历草稿输出：

- 用户回答了第一轮深挖问题；
- 用户补充了岗位方向、案件类型、业务类型、研究题目或量化信息；
- 用户明确要求先直接给一版。

第二轮默认输出：

1. 简短的调整前提说明
2. 定向简历正文
3. 仍待补充的信息

### 第三轮

在以下情况，进入文件交付模式：

- 用户明确要求 Word；
- 用户明确要求 `.docx`；
- 用户要求生成可投递的简历文件。

第三轮默认输出：

1. 简短说明本次文件对应的岗位版本
2. 将最终正文映射为 JSON
3. 生成本地 `.docx` 文件
4. 说明已执行和未执行的校验

## 第一轮结构

### 使用场景判断

至少写清楚：

- 当前材料更适合投什么；
- 为什么这样判断；
- 是否存在场景混杂或场景不清。

### 当前问题

每个问题尽量落到以下类型之一：

- 场景不清；
- 排序不对；
- 经历不贴岗位；
- 法检经历过于事务化；
- 研究缺少业务感；
- 校园经历比重过高；
- 硬技能不可见；
- 缺数字、缺对象、缺产出。

默认控制在 3 到 5 条。

### 调整思路

至少说明：

- 哪些内容删除或后置；
- 哪些内容前置；
- 哪些 bullet 需要改写成动作、场景、产出结构；
- 哪些经历需要补量化；
- 哪些空白需要用法律研究、类案检索或业务研究补位。

默认简写，不做大段展开。

### 进一步内容挖掘问题

默认不要少于 5 个问题，也不要变成通用问卷。

问题应满足：

- 直接围绕用户已有经历；
- 能影响下一版简历；
- 能把经历从单位名称挖成岗位证据。

这部分应是第一轮的主要信息密度来源。建议默认：

- 5 到 8 个问题；
- 先问最影响改写的问题；
- 能对应具体经历就不要抽象发问；
- 如需说明提问目的，用极短短语标注即可。

### 下一步说明

明确说明：

- 用户先补充什么；
- 收到补充后会输出什么；
- 如果用户需要文件，后续会生成 `.docx`。

## 第二轮结构

### 调整前提

先用 2 到 4 句交代：

- 当前版本投什么场景；
- 做了哪些取舍；
- 哪些信息仍属暂定写法。

### 定向简历正文

默认结构：

1. 教育背景
2. 实习经历
3. 法律研究
4. 技能与证书
5. 校园经历（仅在需要时保留）

写作要求：

- bullet 用动作动词开头；
- 每条尽量带场景和产出；
- 同类信息不重复堆叠；
- 法院和检察院经历尽量职业化改写。

### 仍待补充的信息

如果仍有关键缺口，应单列说明：

- 哪几处还缺数字；
- 哪段经历还缺案件或业务细节；
- 哪个研究主题还需要进一步贴业务。

## 特殊情况

如果用户没有成稿简历，第一轮中的“当前问题”可改写为“当前素材短板”，但结构不变。

如果用户明确说“别分析了，先直接给我一版”，可以在第一轮诊断后附暂定草稿，但仍不能跳过场景判断、问题诊断和深挖问题。

## 对 skill 的直接要求

稳定行为应是：

- 先分流；
- 再判断场景；
- 再做问题诊断；
- 再给调整方案和深挖问题；
- 用户补充后再出定向简历；
- 用户要求文件时再交付 `.docx`。
