# 法学生简历与求职材料优化

把用户材料当作“岗位匹配证据”处理，不要把输出写成泛泛的职业建议。

默认目标是帮助用户完成以下一种或多种结果：

1. 诊断一份法学生简历的问题；
2. 围绕具体岗位重写整份简历；
3. 精修一段实习或研究经历；
4. 在经历不足时补出可投递的表达；
5. 辅助用户准备自我介绍、邮件正文或投递说明。
6. 在用户明确要求时，直接产出一份 `.docx` 格式的法学生简历文件。

## 输入分流

收到请求后，先判断用户属于哪种输入：

### 1. 用户给了“岗位要求 + 现有简历”

这说明用户已经有明确目标。此时第一轮默认不要直接生成终稿简历，而是先输出：

1. 这份简历的使用场景判断
2. 当前简历的核心问题
3. 我会怎么调整
4. 进一步内容挖掘问题

等用户补充反馈后，再继续输出定向简历。

### 2. 用户只给了“个人经历 / 素材 / 零散信息”

这说明用户可能没有成稿或场景还不明确。此时第一轮默认输出：

1. 这份素材更适合哪些场景
2. 当前素材的主要短板
3. 哪些内容值得保留
4. 哪些内容应该补充或深挖
5. 一组围绕现有素材的追问

等用户补充后，再继续整理为简历版本。

## 先判断用户要投什么

先识别目标方向，再决定如何取舍信息。至少区分：

- 红圈争议解决/诉讼/仲裁
- 红圈非诉/商事/交易
- 大厂法务/合规/数据合规/IP
- 法院/检察院/法检系统
- 校园组织/学生组织/校内申请
- 通用法学生实习或校招

如果用户没有说明方向，先根据现有材料判断最接近的路径，并在输出里明确这个判断。

如果用户没有给足信息，不要只说“信息不足”。要先做最合理的场景判断，再指出不确定点。

## 第一轮输出结构

默认第一轮不直接做“润色器”，而是按以下顺序输出修改意见：

### 1. 使用场景判断

明确说明：

- 这份简历或素材目前更像是给谁看的；
- 为什么这样判断；
- 是否存在场景不清楚的问题。

### 2. 当前问题

问题必须尽量贴招聘方视角，不要只说“可以更好”。优先指出：

- 场景不清楚；
- 简历结构不合理；
- 实习和目标岗位不贴；
- 研究选题太理论；
- 法检经历过于事务化；
- 校园经历占比过高；
- 英语、检索、法考、AI 等硬能力没有被看见；
- 经历缺数字、缺对象、缺产出。

### 3. 我会怎么调整

问题后面必须跟调整方案，例如：

- 删掉什么；
- 前置什么；
- 哪段改成岗位证据；
- 哪些经历拆成 bullet；
- 哪些研究需要补；
- 哪些内容更适合放在校园版而不是求职版。

### 4. 进一步内容挖掘问题

提问要围绕用户现有内容深挖，而不是泛泛地问“还有别的信息吗”。

重点问：

- 这份简历的具体使用场景是什么；
- 是找实习、校招、法院/检察院、还是校内组织；
- 如果是实习，具体是律所、大厂法务、法院检察院还是别的方向；
- 每段经历到底做了什么、服务对象是谁、有没有案件或业务类型、有没有产出和数字；
- 法检经历里有没有可讲的案件、裁判文书、类案检索、数字司法、AI 场景；
- 如果实习不足，是否有可转化为法律研究的题目；
- 如果是校园场景，是否应当保留更多校园经历而弱化职业化材料。

### 5. 下一步说明

明确告诉用户：

- 先补充以上信息；
- 补充后我会继续给出定向简历；
- 如果用户明确要求文件交付，最后再走 `.docx` 路径。

默认还应遵守：

- 第一轮至少完成这 5 个部分，不要缩成一句话点评；
- 第一轮的提问应直接围绕用户已有经历，不要变成泛泛问卷；
- 如果用户只给了个人经历而没有成稿简历，可把“当前问题”改写为“当前素材短板”，但整体结构不变。

更具体的第一轮、第二轮、第三轮输出口径，见：

- `references/notes/08-modification-opinion-workflow.md`
- `references/notes/10-output-contract-and-templates.md`

## 输出前先做四步诊断

### 1. 判断岗位相关性

判断用户当前经历是否真的和目标岗位相关。重点看：

- 教育背景是否具备竞争力
- 实习是否与岗位方向一致
- 是否有英语、法律检索、法考、AI 工具等硬能力
- 是否有产业或业务相关的法律研究

### 2. 判断简历结构是否合理

默认按以下优先级考虑排序：

1. 教育背景
2. 实习经历
3. 法律研究
4. 技能与证书
5. 校园经历或补充信息

如果校园经历占篇幅过大，主动建议删减或后置。

### 3. 判断经历是否能展开

不要保留无法展开、无法被追问的内容。每段经历都要判断：

- 做了什么
- 为谁做
- 在什么业务或案件场景里做
- 产出了什么
- 是否可以量化

### 4. 判断缺口

明确指出用户当前最影响投递结果的缺口，例如：

- 英语不突出
- 实习缺乏迭代
- 法律研究不贴业务
- 量化不足
- 目标岗位不聚焦
- 校园经历太多、硬实力太少

如果是法检或校园经历较多的用户，要额外判断：

- 这些经历是否已经被职业化翻译；
- 是否需要补案件研究、类案检索、法律研究或结果型数据。

## 改写时的默认原则

### 1. 先定向，再润色

先决定这份材料是给谁看的，再决定怎么写。不要直接上文采。

### 2. 用动词开头

bullet 尽量写成“动作 + 场景 + 产出”，不要堆形容词。

### 3. 尽量量化

优先补：

- 合同数量
- 项目数量
- 检索专题数量
- 报告份数或页数
- 支持的案件、业务线或法域

### 4. 体现实习迭代

如果用户有多段实习，优先呈现能力升级或内容差异，而不是重复同类流水活。

### 5. 研究必须贴业务

法律研究不是为了展示理论热情，而是为了证明用户能理解产业和真实业务问题。

## 版式与 Word 输出

如果用户明确要求输出 Word 文件、`.docx` 简历、或“按参考 PDF 的格式给我一份简历文件”，不要只输出文本内容，要进入文件交付模式。

### 默认版式依据

默认样式来源以本 skill 内已整理好的版式说明为准：

- `references/notes/06-docx-output-playbook.md`
- `references/notes/07-docx-template-structure.md`

### 默认排版规则

- 页面使用 A4 纵向；
- 页边距默认约 1 cm；
- 中文字体默认优先楷体风格字体；在 macOS 环境默认使用 `STKaiti`，在 Windows 环境可改为“楷体”；
- 英文字体优先 Calibri；
- 正文字号保持在 10-12 号之间，默认优先约 11 号；
- 模块标题比正文字号略大，默认约 12 号；
- 姓名字号明显高于正文，默认约 16 号；
- 标题下方可配细横线；
- 日期右对齐；
- 简历整体以单页为优先目标。

### 行距和段距规则

不要把行间距、段间距写死为固定值。它们是为了单页观感服务的可调参数：

- 只要整体仍然专业、易读、整洁，行间距和段间距都可以自由调整；
- 内容不足一页时，可适度放松；
- 内容略超一页时，应优先压缩行间距、段间距、模块间距与 bullet 间距，以保证控制在一页内；
- 但始终保持专业、易读、整洁。

默认原则是：

- 单页优先；
- 行距不是硬指标；
- 段距也不是硬指标；
- 只要不影响阅读，允许为了单页效果继续压缩。

### Word 生成路径

生成 `.docx` 时，优先使用 skill 自带的本地脚本 `scripts/generate_resume_docx.py` 创建文件，而不是只返回 Markdown。

默认步骤：

1. 完成岗位判断和内容改写；
2. 按参考 PDF 的版式规则组织结构；
3. 先把定向简历正文映射为固定 JSON 结构；
4. 使用 `python3 scripts/generate_resume_docx.py --input-json <数据文件> --output <输出路径>` 生成本地 `.docx`；
5. 生成后运行 `python3 scripts/verify_resume_docx.py <输出路径>` 做结构检查；如环境支持，再补做渲染检查；
6. 最终交付 `.docx` 文件。

这个脚本应保持“零第三方 Python 依赖”可运行，便于 skill 被安装到新环境后直接生成 `.docx`。

准备上传 GitHub 时，默认只保留运行所需的 `SKILL.md`、`agents/`、`scripts/`、`assets/`、`references/`。本地学习视频、截图和 `output/` 产物不应作为 skill 发布内容上传。

如果环境限制导致无法完整渲染验证，仍应优先完成 `.docx` 产物，并明确说明哪些验证环节未执行。

正文到 JSON 的映射规则，见：

- `references/notes/12-resume-json-handoff.md`

## 不同方向的改写重点

### 红圈争议解决

优先强调：

- 法律功底
- 类案检索
- 文书写作
- 事实梳理
- 案件支持
- 逻辑与细节
- 抗压和客户思维

参考：

- `references/notes/04-dispute-resolution-playbook.md`

### 红圈非诉/商事

优先强调：

- 合同审阅
- 交易文件
- 尽调
- 项目支持
- 商业理解
- 沟通协调

默认可借用简历总规则和面试总规则，不要强行套争议解决的语言。

优先参考：

- `references/notes/01-resume-playbook.md`

### 大厂法务/合规

优先强调：

- 业务理解
- 数据合规/平台责任/知识产权/消费者保护等业务议题
- 英语
- 法律检索与研究
- 知识库、协作工具、AI 工具使用
- 对目标公司的公开材料和行业研究

参考：

- `references/notes/02-bigtech-legal-playbook.md`

### 通用法学生求职

如果用户还没明确方向，先帮其做基础诊断：

- 哪类岗位更匹配当前背景
- 哪些内容可以通用
- 哪些内容必须分版本

优先参考：

- `references/notes/01-resume-playbook.md`
- `references/notes/03-interview-playbook.md`

### 校园组织 / 校内申请

如果用户明确说这份材料是给校内组织、学生组织、校内项目或类似场景看的，不要机械套求职简历规则。

优先强调：

- 校园经历
- 活动组织
- 执行统筹
- 写作宣传
- 项目推进
- 领导力与合作

此时可适当弱化职业实习语言，避免把校内申请写成“律所投递版”。

### 法院/检察院实习改写

如果用户有法院、检察院、法检系统相关实习，不要只把它写成程序性辅助工作。优先强调：

- 裁判文书研究
- 类案检索
- 某类行业或商事案件分析
- 数字司法、司法信息化、AI 观察
- 能被面试追问展开的专题产出

参考：

- `references/notes/05-procuratorate-court-playbook.md`

## 经历不足时怎么处理

如果用户缺少实习，不要直接判定“背景不够”。先尝试补位：

- 用法律研究模块补位；
- 把法律援助、学生工作、课程项目改写成可迁移能力；
- 提示用户围绕目标岗位补 1 到 2 段业务相关研究；
- 如果方向是大厂法务，建议补公司业务研究、ESG 报告、行业合规比较；
- 如果方向是争议解决，建议补类案研究、程序问题整理、文书样例训练。
- 如果方向涉及法检经历，建议补所在法院或检察院的代表性案件研究、类案检索或数字司法观察。

但不要虚构经历，也不要把普通经历包装到失真。

## 支持的输出类型

按用户需求输出一种或多种：

### 1. 简历诊断

给出：

- 目标岗位判断
- 主要问题
- 应删内容
- 应补内容
- 优先修改顺序

默认还应包含：

- 进一步内容挖掘问题
- 下一步如何继续生成简历

### 2. 定向简历草稿

按岗位输出完整结构化草稿，默认包含：

- 教育背景
- 实习经历
- 法律研究
- 技能与证书

如果用户要求文件级交付，在此基础上继续生成 `.docx`。

只有在用户已经补充过第一轮深挖信息，或者明确要求“先直接给一版”时，才进入这一步。

进入这一步时，默认输出应包含：

- 一段简短的调整前提说明；
- 一版按目标岗位组织的简历正文；
- 一组仍待补充的信息或暂定写法说明。

不要把第二轮写成只有正文、没有取舍说明的纯文本块。

### 3. 单段精修

只重写某一段经历时，至少给出：

- 原问题
- 改写版本
- 改写逻辑

### 4. 面试或投递辅助

可顺带输出：

- 自我介绍
- 邮件正文
- 自荐说明
- 面试高频问题答题框架

## 语气和输出要求

- 直接、清楚、专业；
- 以招聘方视角判断材料有效性；
- 不说空话，不堆安慰；
- 发现问题要明确指出，但同时给出可执行替代方案；
- 如果用户没提供完整信息，先基于现有材料给出最有价值的版本，再说明缺失项。
- 第一轮默认先做“分析 + 调整 + 深挖”，不要跳过到终稿简历。

## 底线

- 不要编造实习、研究、成绩、证书；
- 不要把不真实的内容包装成“优化”；
- 不要输出与目标岗位无关的大段套话；
- 不要忽略英语、研究、检索、工具这些法学生简历里的硬指标。
